## 1. Brand Identity

### Brand Philosophy

**aaraazi** is a comprehensive real estate management platform built for the Pakistani market (Karachi). The brand reflects:

- **Professional Excellence**: ERP-standard real estate operations
- **Trust & Reliability**: Secure, data-driven decision making
- **Modern & Approachable**: Warm, human-centered design
- **Growth & Success**: Tools that help businesses thrive

### Brand Personality

| Trait | Description |
|-------|-------------|
| **Professional** | Enterprise-grade, trusted by agencies |
| **Warm** | Human-centered, approachable interface |
| **Modern** | Contemporary design, up-to-date patterns |
| **Trustworthy** | Secure, reliable, data-focused |
| **Growth-Oriented** | Success metrics, analytics, insights |

### Brand Positioning

- **Market**: Pakistani real estate agencies (Karachi-focused)
- **Target**: Professional agencies, developers, brokers
- **Value Prop**: Complete ERP-standard real estate management
- **Differentiator**: Pakistani market-specific (PKR, local practices)

## 3. Typography

### Font Family

**Primary**: **Inter** (Google Fonts)
- Modern, readable, professional
- Excellent for data-dense ERP applications
- Great legibility at all sizes

**Fallback**: `system-ui, -apple-system, sans-serif`

### Font Weights

| Weight | Value | Use |
|--------|-------|-----|
| **Light** | 300 | Large decorative text (rare) |
| **Regular** | 400 | Body text, descriptions |
| **Medium** | 500 | Labels, emphasized text |
| **Semibold** | 600 | Headings, important text |
| **Bold** | 700 | Strong emphasis (rare) |

### Type Scale

| Name | Size | Line Height | Use |
|------|------|-------------|-----|
| **xs** | 10.5px | 14px | Fine print, metadata |
| **sm** | 12px | 16px | Labels, captions |
| **base** | 14px | 20px | Body text (default) |
| **lg** | 16px | 24px | Emphasized body |
| **xl** | 18px | 28px | Small headings |
| **2xl** | 24px | 32px | Section headings |
| **3xl** | 30px | 36px | Page headings |
| **4xl** | 36px | 40px | Hero headings |
| **5xl** | 42px | 48px | Display (rare) |

### Typography Usage

#### Headings
```tsx
// âœ… GOOD - Let CSS handle sizing
<h1>Property Details</h1>  // Uses text-3xl from globals.css
<h2>Overview</h2>           // Uses text-2xl from globals.css
<h3>Description</h3>        // Uses text-xl from globals.css
```

#### Body Text
```tsx
// âœ… GOOD - Default is already 14px
<p>This is a description of the property...</p>

// âœ… GOOD - Only override when needed
<p className="text-sm text-slate-500">Secondary information</p>
```

#### Labels & Captions
```tsx
// âœ… GOOD - Small text for labels
<label className="text-sm font-medium">Property Type</label>

// âœ… GOOD - Extra small for metadata
<span className="text-xs text-slate-400">Updated 2 hours ago</span>
```

### Typography Rules

**âŒ DON'T:**
- Use Tailwind typography classes unnecessarily (`text-xl`, `font-bold`)
- Override heading sizes unless absolutely needed
- Use more than 3 font weights in a single view
- Go below 12px for body text (accessibility)

**âœ… DO:**
- Let CSS handle default typography
- Use semantic HTML (`<h1>`, `<h2>`, `<p>`)
- Maintain consistent hierarchy
- Test readability at all viewport sizes

## 5. Design Principles

### Core Philosophy

> "Every page should feel familiar, every interaction should be predictable, every component should be accessible. **But when unique contexts demand unique solutions, the system should flex to support them.**"

### Design System Mindset

**Consistency â‰  Rigidity**

The design system provides:
- **Strong defaults** for 90% of use cases (use the core templates)
- **Flexibility** for the 10% that need something different
- **Guidance** on when to follow patterns vs when to innovate
- **Quality standards** that apply to all components

### When to Reuse vs Create

| Fit | Action | Example |
|-----|--------|---------|
| **80-100%** | âœ… **REUSE** | Use existing template as-is |
| **50-80%** | âš ï¸ **EXTEND** | Add props, customize template |
| **<50%** | ğŸ†• **CREATE** | New template, document why |

### Template Decision Tree

1. Does **DetailPageTemplate** fit? â†’ Use it
2. Does **WorkspaceTemplate** fit? â†’ Use it
3. Does **FormTemplate** fit? â†’ Use it
4. Can you **extend** one of these? â†’ Extend it
5. Does an **existing custom template** fit? â†’ Use it
6. Do you **need a new template**? â†’ Create it, document why

### Quality Standards (All Components)

âœ… **Mandatory for ALL**:
- **UX Laws**: Fitts, Miller, Hick, Jakob, Aesthetic-Usability
- **Design Tokens**: Use CSS variables, 8px grid, no arbitrary values
- **Accessibility**: WCAG 2.1 AA, keyboard nav, ARIA labels
- **TypeScript**: Full type definitions
- **Responsive**: Mobile, tablet, desktop
- **Documentation**: JSDoc with examples

## 7. UX Laws Implementation

### 1. Fitts's Law (Targeting)

**Principle**: The time to acquire a target is a function of distance to and size of the target.

**Implementation**:
```tsx
// âœ… GOOD - Large primary button (easy to click)
<button className="px-6 py-3 text-base">  // 44px+ min height
  Add Property
</button>

// âœ… GOOD - Optimal placement (top-right corner)
<div className="flex justify-end">
  <button>Primary Action</button>
</div>

// âŒ BAD - Small, hard to click
<button className="px-2 py-1 text-xs">Add</button>
```

**Rules**:
- Primary buttons: Minimum 44x44px touch target
- Place important actions in easy-to-reach locations
- Increase size for frequently used actions

### 2. Miller's Law (Cognitive Load)

**Principle**: The average person can hold 7Â±2 items in working memory.

**Implementation**:
```tsx
// âœ… GOOD - Max 5 metrics (within 7Â±2)
<PageHeader
  metrics={[
    { label: 'Revenue', value: 'PKR 25M' },
    { label: 'Deals', value: '12' },
    { label: 'Leads', value: '45' },
    { label: 'Conversion', value: '18%' },
    { label: 'Properties', value: '25' }
  ]}
/>

// âŒ BAD - Too many items (cognitive overload)
<PageHeader
  metrics={[...15 different metrics]}
/>
```

**Rules**:
- Limit to 5-7 items per group
- Break long lists into categories
- Use progressive disclosure for complexity

### 3. Hick's Law (Decision Time)

**Principle**: The time to make a decision increases with the number of choices.

**Implementation**:
```tsx
// âœ… GOOD - Limited primary actions, overflow in dropdown
<PageHeader
  primaryActions={[
    { label: 'Edit', onClick: handleEdit },
    { label: 'Share', onClick: handleShare }
  ]}
  secondaryActions={[
    { label: 'Export', onClick: handleExport },
    { label: 'Archive', onClick: handleArchive },
    { label: 'Delete', onClick: handleDelete }
  ]}
/>

// âŒ BAD - Too many visible choices
<div className="flex gap-2">
  <button>Edit</button>
  <button>Share</button>
  <button>Export</button>
  <button>Archive</button>
  <button>Delete</button>
  <button>Duplicate</button>
  <button>Print</button>
  <button>Email</button>
</div>
```

**Rules**:
- Max 3 primary actions visible
- Group related actions together
- Use progressive disclosure (dropdowns, modals)

### 4. Jakob's Law (Familiarity)

**Principle**: Users prefer your site to work the same way as all the other sites they already know.

**Implementation**:
```tsx
// âœ… GOOD - Standard breadcrumbs (top-left)
<div className="flex items-center gap-2 mb-4">
  <button onClick={handleBack}>â†</button>
  <span>Properties</span> / <span>Modern Villa</span>
</div>

// âœ… GOOD - Standard action placement (top-right)
<div className="flex justify-between items-center">
  <h1>Title</h1>
  <div className="flex gap-2">
    <button>Cancel</button>
    <button>Save</button>
  </div>
</div>
```

**Rules**:
- Follow platform conventions (iOS, Android, Web)
- Use standard patterns (breadcrumbs, pagination, search)
- Don't reinvent common UI patterns

### 5. Aesthetic-Usability Effect

**Principle**: Users perceive aesthetically pleasing designs as more usable.

**Implementation**:
```tsx
// âœ… GOOD - Clean, professional card
<div className="bg-white rounded-lg p-6 shadow-sm space-y-4">
  <h3 className="font-semibold text-lg">Property Name</h3>
  <p className="text-slate-600">Description</p>
  <div className="flex gap-4">
    <MetricCard label="Price" value="PKR 75M" />
    <MetricCard label="Area" value="500 sq yd" />
  </div>
</div>

// âŒ BAD - Cramped, cluttered
<div className="p-2">
  <h3>Property Name</h3><p>Description</p>
  <span>PKR 75M</span><span>500 sq yd</span>
</div>
```

**Rules**:
- Generous spacing (8px grid)
- Consistent alignment
- Visual hierarchy
- Professional polish
